## engine core (40 sites)

**Summary:** Engine-core sweep, 40 sites. Structure: ALL Y flows through ONE bake — constraints.ts:18 wcagY->oklchToLinearRgb (sRGB matrix + Math.max(0,ch) crush) — plus 3 parallel bakes: apcaY sRGB coefficients (constraints.ts:53, consumes makeStop rgb), apparentL's sRGB->XYZ matrix (perceptualL.ts:7), and clampChromaToGamut's sRGB cube (constraints.ts:82). Parameterize those 4 and everything downstream follows. render-gamut-Y (must track P3): the whole H-K stack (apparentL/solveLForApparent/solveCForApparent/meanBoost/perceptualRungL/perceptualDarkC -> darkChromaCurve) — NOTE meanBoost's sRGB ceiling truncation IS the measured gold-band block behind C7; illustration Y-equalization (colorEngine 199/200/204); on-text pole preference (onTextIsWhite apcaLc + apcaY call sites colorEngine 241/298); collision deltaE gates (collision.ts 50/98 — INDIRECT via gamut-clamped ColorStop.C; verdicts can flip under P3 -> remedy changes -> re-bless). legal-sRGB-question (owner must rule sRGB-clamp vs rendered): findLForContrast/findMaxLForContrast bodies, applyRedRepelRender 4.5 gate+re-solve (colorEngine 182/184), onTextIsWhite enforce-flip + ratioFloor (colorMath 199/207), stopTable S8/T11/T12 ratio targets. gamut-clamp (becomes gamut-parameterized): clampChromaToGamut, maxChromaAt (pastelK ceiling — owner eye-check, P3 pastels go more chromatic), makeStop's clamp = THE master fork; every makeStop mint site listed. emit-boundary: oklchToSrgbUnclamped + makeStop rgb field — today ONE field conflates rendered color and sRGB emit; P3-master needs stop to carry rendered channels + separate sRGB clamp-down at emit, else apcaY's 0..1 clamp silently crushes P3-legal colors. Neutral: checkHueCollision TYPE-1 gate (hue+pre-clamp brandC — swap decisions gamut-stable), grayApparentL (C=0), gray seed round-trip, neutralCurve.ts entirely (pure interpolation, absolute peak C<=0.02 in-gamut both — zero call sites, none reported), stopTable scaffold constants, darkCtaTrim hue lobes. gate-instrument: none in engine-core (audit scripts outside area), but flagged: C7 sweep's 0.006 washDeltaE bar and highlight-audit S8 gate both move under P3 -> one-shot re-bless as planned. Caches: perceptualL boostCache key lacks gamut — must include it if both gamuts resolve in one process.

## Appendix: Figma Display-P3 verification (kickoff step 3)

FINDINGS (2026-07 current):

**(1) Doc profile affects RGBA interpretation? YES — profile-relative, not absolute.**
Plugin RGB/RGBA doc: "Figma supports color management. All colors are specified in the same color space which will be the color profile of the document (documentColorProfile)." Same `{r,g,b}` 0–1 triple = different physical color in sRGB doc vs P3 doc. Variables inherit this: COLOR variable value is plain RGB/RGBA, so a variable's numbers mean "coordinates in whatever profile the consuming file has."
→ https://www.figma.com/plugin-docs/api/RGB (redirects to developers.figma.com/docs/plugins/api/RGB/)

**(2) Plugin read/set doc profile? READ yes, SET no.**
`figma.root.documentColorProfile: 'LEGACY' | 'SRGB' | 'DISPLAY_P3'` — **[readonly]**. "LEGACY" = docs predating color management (Unmanaged deprecated Aug 2023, now treated ~sRGB/preferred). No plugin or REST API changes file profile; user-only via File → File color profile. Export is separately controllable: `ExportSettings.colorProfile: 'DOCUMENT' | 'SRGB' | 'DISPLAY_P3_V4'`.
→ https://developers.figma.com/docs/plugins/api/DocumentNode/ ; https://developers.figma.com/docs/plugins/api/ExportSettings/

**(3) Gamut-tagged storage? NO.**
`VariableValue = string | number | boolean | RGB | RGBA | VariableAlias` — raw untagged 0–1 components, zero profile/gamut field. Value is interpreted in doc profile at render time.
→ https://www.figma.com/plugin-docs/api/VariableValue (developers.figma.com/docs/plugins/api/VariableValue/)

**(4) REST/plugin Variables API out-of-sRGB or wide-gamut declaration? NO declaration mechanism.**
REST POST `/v1/files/:key/variables` COLOR value = `{"r":0.2,"g":0.2,"b":0.2,"a":1}`; docs specify only "value must match the variable's type," no profile/gamut param anywhere. RGB type contract is 0–1; components >1 or <0 are out-of-spec (expect rejection/undefined behavior — not documented as clamping). Only way to express P3-only colors: put the FILE in Display P3 profile and encode them as in-range 0–1 P3 coordinates. Cannot mark a single variable wide-gamut.
→ https://developers.figma.com/docs/rest-api/variables-endpoints/ ; https://www.figma.com/plugin-docs/working-with-variables/

**(5) Paste/import P3 values into sRGB doc? RAW VALUES KEPT, REINTERPRETED — no conversion.**
Bjango (documenting Figma behavior): "copying and pasting properties between sRGB and Display P3 documents doesn't convert colours." So P3 coordinates pasted into sRGB doc render as sRGB coordinates (appearance shifts/desaturates vs intent); reverse direction over-saturates. Whole-file profile switch is the only converting path: **Assign** = keep values, appearance changes; **Convert** = keep appearance, values change, and "converting a file from Display P3 to sRGB may 'clip' some colors."
→ https://bjango.com/articles/colourmanagementsettings/ ; https://help.figma.com/hc/en-us/articles/360039825114-Manage-color-profiles-in-design-files

**Extra signals:**
- Dev-mode `getCSSAsync()` emits doc-profile values untagged; developer bug report (Apr 2025) on P3/sRGB mismatch got no Figma reply: https://forum.figma.com/report-a-problem-6/when-developing-plugins-i-found-two-display-p3-color-issues-39211
- Browser Figma historically renders sRGB-limited; full P3 needs desktop app + P3 display (forum/help, e.g. https://forum.figma.com/t/launched-color-profiles/1179).

**Implication for P3 master-gamut plan:** Figma variables are profile-RELATIVE. Plugin must branch on `figma.root.documentColorProfile`: doc=DISPLAY_P3 → write P3 coordinates directly (0–1, full P3 gamut usable); doc=SRGB/LEGACY → must gamut-map/clamp to sRGB coordinates before writing (matches clamp-DOWN-to-sRGB direction in docs/engine-spec/P3-KICKOFF.md). Cannot programmatically flip doc to P3; cannot store both gamuts in one variable except via separate modes/collections (values still profile-relative per consuming file).
