## 1. Visual design direction

### 1a. highlight uses the WRONG MATH — anchor hue instead of ladder hue (the main bug)

Conceptual model (corrected): **`cta` is the brand anchor** — the brand's natural
lightness + raw brand hue, the literal brand-colored fill. `identity` is the exact
input hex. **`highlight` is a pure scale position** and should have NO relationship
to the anchor; it must behave like any other ladder step.

The bug: `highlight-9/10` is currently generated by `brandHighlightPair` as the
fill of a **vivid-forced `generateScale` pass** — i.e. derived from the *anchor*
computation, which by design uses the **raw `brandH`** and skips the light ramp's
warm-torsion hue drift (the gold-spine rotation that warms every other stop as it
darkens). So highlight wrongly inherits *anchor* behavior (raw hue) when it should
have *ladder* behavior (drifted hue at its lightness).

Measured hue (H) — the ramp drifts warm down the scale, but highlight doesn't:

| hue | brandH | accent-8 H | highlight-9 H | stamp-11 H | ink-12 H |
|---|---|---|---|---|---|
| yellow | 84.1 | 75.5 | **84.1** | 63.8 | 62.8 |
| orange | 41.7 | 41.4 | **41.7** | 39.1 | 38.8 |
| blue/pink | — | (≈brand) | (≈brand) | (≈brand) | — |

highlight sits at the raw 84.1 (cooler yellow-green) while its darker neighbor
stamp-11 has drifted to 63.8 (warm gold) — so **highlight reads cooler than 11**,
breaking the ramp. Only warm hues (in the gold-spine band) show it; blue/pink
don't drift so they look fine.

**Fix (mechanism, not a constant):** compute highlight like a ladder stop
(1–8 / 11 / 12) — `lightHueAt(targetL)` for hue + the ladder/envelope chroma
model — at a fixed target L, instead of pulling the fill of a forced-archetype pass.
That machinery is local to `generateScale`, so the clean fix computes the
highlight pair *inside* `generateScale` (light + dark, using `lightHueAt` and the
dark `torsionedHue`) and attaches it to the scale; `brandHighlightPair` /
`resolveBrand`'s separate pass goes away.

> **Engine-constraint caveat — read before implementing.** This fix edits the
> CORE `generateScale` computation path, which is past the "additive + renaming,
> byte-identical existing hexes" line the shipped work stayed within. It was
> deliberately NOT done as part of that work. Treat it as a scoped engine change:
> make the design decisions in §1b/§2 first, then implement, and `npm run audit`
> MUST still report a byte-identical snapshot afterward (the change only adds the
> highlight pair — it must not move any existing stop). Do not bundle it into an
> unrelated edit.

### 1b. highlight sits too close to stamp-11 (L target)

Even with the hue fixed, the **L target** needs choosing. Today's pin (vivid
archetype, L 0.60) sits near the bottom of the accent-8↔stamp-11 window, so
highlight crowds the text stop and the hover dips below it. Measured light-mode L:

| hue | accent-8 | highlight-9 | highlight-10 | stamp-11 |
|---|---|---|---|---|
| pink | 0.738 | 0.591 | 0.547 | 0.530 |
| blue | 0.738 | 0.574 | 0.529 | 0.530 |
| yellow | 0.766 | 0.564 | **0.519** | **0.558** |

- yellow: `highlight-10` (0.519) is **darker** than `stamp-11` (0.558) — inversion.
- Causes: hover steps darker, and the yellow brightness-lift raises stamp-11.

**Decisions needed:**
- Target L for highlight — likely **~0.65** (midpoint of accent-8 ≈ 0.74 and
  stamp-11 ≈ 0.53) rather than 0.60. With the ladder-hue fix this is just a
  constant in `generateScale`, no archetype needed.
- Hover direction: should `highlight-10` go *lighter* than `highlight-9`?
- Ensure stamp-11 (incl. yellow-lift) never crosses highlight.

## 2. Text color allowed on highlight

Today `brandHighlightPair` runs through the brand floor with
`enforceOnFillContrast: true`, so highlight inherits the engine's APCA/WCAG
on-fill polarity and **darkens itself** when white text would fail. That
darkening is part of why highlight drops toward stamp-11 (§1a).

**Decisions needed:**
- Is `highlight` a fill that must carry legible text? If so, which polarity —
  white only, black only, or either (and does it get its own `on-highlight`
  token)?
- If highlight is allowed **black text**, it can stay lighter (no contrast
  darkening), which also helps §1a. If it must hold **white text**, light hues
  (yellow) will keep darkening — accept that or pin L regardless of contrast.
- Same question for neutral `cta` in dark mode (near-white fill ⇒ dark text);
  decide whether `cta` needs its own on-fill token distinct from the ramp's.

## 3. Demo control — show each generated scale

The demo currently renders the **brand** ramp only. Add a control (segmented or
dropdown) to switch the annotated card between ramps:
`brand · secondary · neutral · error · warning · success · info`.

- Each ramp's token set differs: brand/secondary = 15 (incl. `identity`,
  `cta`/`cta-hover`); neutral = 14 (`cta` pair, no `identity`); signals = 12
  (no `cta`, no `identity`). The card should add/drop the relevant annotated
  elements per ramp.
- The token values are already available: brand/secondary via
  `resolveBrand(hex).scale` (+ `.highlight`, `.identityHex`); neutral via
  `generateNeutralScale` + `neutralCtaPair`; signals via `SIGNAL_SCALES`. CSS for
  all of them is already emitted globally (`dist/neutral.css`, `dist/signals.css`)
  and per-brand via `brandCss`, so the card can reference `var(--<ramp>-<token>)`.
- This control is also the validation surface for §1 — switching to each ramp at
  several hues is how the highlight-ordering fix gets eyeballed.
